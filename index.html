    Identación

    Normalmente se puede trabajar con identación y espacios en blanco en el código y no es importante para el intérprete puesto que no los toma en cuenta, pero para el programador, esto provee una mejor legibilidad del código.

    Entonces la recomendación es que:

  • Nunca se debe mezclar espacios y tabulaciones.
  • De preferencia, se debe identar con 4 espacios.
  • No uses tabulaciones
    • Longitud de una lí­nea

    El lí­mite de las lí­neas de código deberán ser de 80 caracteres, esto ayudará a una mejor lectura del código.

    Llaves

    La llave de apertura deberá ir en la misma lí­nea de la sentencia. Si colocas en la siguiente lí­nea, en algunos casos muy particulares se puede producir algún error. Más adelante se explica dicho error en el uso de punto y coma en las sentencias.

    No recomendado

    // control flow stament if ( true ) { //codes goes here } //Anonymous function declaration function ( args ) { return true; } //Named function declaration function foo() { return true; } //Anonymous function expression const bar = function ( args ) { return true; } //Arrow function const bar = (arg) => { return true, }; const bar = ( args ) => { true };

    Recomendado

    // control flow stament if ( true ) { //codes goes here } //Anonymous function declaration function ( args ) { return true; } //Named function declaration function foo() { return true; } //Anonymous function expression const bar = function ( args ) { return true; } //Arrow function const bar = (arg) => { return true, }; const bar = ( args ) => { true }; const bar = ( args ) => true;
    Declaración de variables

    En el ámbito de las variables y hoisting, ahora es recomendable usar const y let en lugar de var.

    Formato de declaración

    No Recomendado

    // separar por comas const day = ‘Monday’, count = 10; //Alinear const day = ‘Monday’, count = 10; //En la misma lí­nea const day = ‘Monday’, count = 10;

    Recomendado

    const day = 1; const count = 10; const keys = [‘foo’, ‘bar’]; const values = [1, 2];
    Formato de nombres de variables y funciones
  • Utiliza camelCase para nombrar funciones, declaración de variables, instancias, etc.

    • Por ejemplo:

    - functionNamesLikeThis; - variableNamesLikeThis; - methodNamesLikeThis;
  • Utiliza PascalCase para nombrar constructores, prototypes, clases, etc

    • Por ejemplo:

    - ConstructorNamesLikeThis; - EnumNamesLikeThis; En variables
  • Para declarar valores constantes utiliza Mayusculas y separado por un guión bajo cada palabra.

    • Por ejemplo:

    const SYMBOLIC_CONSTANTS_LIKE_THIS;
  • Para declarar valores variables utiliza camelCase

    • Por ejemplo:

    let adminUser; let daysSinceCreation;

    Nombres y comentarios en ingles

    En general, los lenguajes de programación basan su vocabulario en el idioma inglés, puesto que es el lenguaje más utilizado en el mundo. Se debe escribir los comentarios, como nombres de variables, de funciones y demás, en ingles.

    Nombres con sentido

    La intención de asignar un nombre con sentido a carpetas, archivos, variables, funciones, etc. Es indicar cuál es su cometido, de tal manera que quede claro el propósito por el cuál fue creado.

    Comentarios

    Procura asignar comentarios solo cuando sea necesario y que ayuden a entender mejor el código.

    Nombres de archivo

    La mayoría de los servidores web (Apache, Unix) son sensibles acerca de los nombres de archivo: london.jpg no se puede acceder como London.jpg . Otros servidores web (Microsoft, IIS) no distinguen entre mayúsculas y minúsculas: london.jpg se puede acceder como London.jpg o london.jpg. Si se utiliza una mezcla de letras mayúsculas y minúsculas, usted tiene que ser muy consistente. Si se mueve de un caso insensible, a un servidor de mayúsculas y minúsculas, incluso los pequeños errores pueden romper su sitio web. Para evitar estos problemas, utilice siempre los nombres de archivo en minúsculas (si es posible).

    • A tener en cuenta

  • Si se crea una nueva página, esta deberá tener sus propios archivos de estilos.
  • Los archivos se nombraran en ingles y en minuscula
  • Los archivos estarán en formato .scss
  • El archivo que contiene cada grupo de estilos debera tener un _index.scss para agruparlos todos y tener una organización mas adecuada.
  • Se debe usar los import correctamente para que los estilos se apliquen correctamente.
  • Los archivos deberán estar nomprados por el tipo de estilo, ejemplo, si los estilos son de las moléculas de la página home, se nombrara así: _molecule-home
